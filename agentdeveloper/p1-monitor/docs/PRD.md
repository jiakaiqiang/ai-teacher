## 1. 产品定位

### 1.1 一句话描述
一个能够 **实时监控工业设备指标、自动检测异常、并由 AI Agent 自主完成诊断与决策** 的智能监控平台。

### 1.2 为什么做这个产品

#### 真实痛点
```
传统监控系统的问题：
  ❌ 告警泛滥：阈值告警太多，90% 是噪音
  ❌ 人工诊断：每次异常需要人工查看多个系统、分析日志
  ❌ 响应延迟：从异常发生到处置决策平均 30-60 分钟
  ❌ 知识孤岛：相似问题重复诊断，经验无法复用
```

#### 本产品的价值
```
智能监控平台的优势：
  ✅ 智能检测：动态基线 + 趋势分析，误报率 < 10%
  ✅ 自主诊断：Agent 自动完成 5 步诊断，平均耗时 < 30s
  ✅ 闭环决策：自动判断处置方式（通知/工单/忽略）
  ✅ 知识沉淀：每次诊断记录可供后续检索
```

### 1.3 目标用户

- **运维工程师**（70%）：日常监控、异常处置的执行者
- **SRE 工程师**（20%）：系统稳定性负责人
- **技术管理者**（10%）：查看运维质量报表

### 1.4 使用场景

#### 场景 1：CPU 突增异常（高频）
```
09:23:15  系统检测到 srv-001 CPU 95% 持续 30s
09:23:18  Agent 自动唤醒开始诊断
         ├ 调用 getServerHealth → 确认当前 96%
         ├ 调用 getProcessList → 发现 java 进程占 80%
         ├ 调用 queryHistoricalAnomalies → 找到 3 条相似案例
         ├ 生成诊断报告："Java 应用异常占用，建议重启"
         └ 调用 createTicket(severity='medium')
09:23:28  运维工程师收到工单通知
09:25:00  处理完成，Agent 记录解决方案
```

#### 场景 2：温度异常（关键）
```
14:56:10  检测到 srv-003 温度 35°C（阈值 32°C）
14:56:15  Agent 诊断
         ├ 调用 getRoomTemperature → 机房温度正常
         ├ 调用 getServerHealth → 风扇转速异常
         ├ 判断为硬件故障
         └ 创建紧急工单 + 发送告警通知
14:56:20  运维立即响应
```

#### 场景 3：用户主动询问（低频）
```
用户在 Chat 浮层输入：
  "过去 1 小时哪些机器最不稳定？"

Agent 响应：
  ├ 调用 listAnomalies(hours=1)
  ├ 调用 getServerHealth(批量)
  ├ 统计分析
  └ 返回："srv-001 有 3 次 CPU 突增，srv-005 有 2 次内存告警..."
```

**交互规则**

2. 异常点在图表上标红色 markPoint

4. 支持切换设备、切换时间范围

**性能要求**
- 渲染帧率：60fps（Echarts 优化）
- WebSocket 延迟：< 100ms
- 支持同时监控 ≥ 10 台设备

#### 2.2.2 异常检测引擎

**检测算法**

| 算法类型 | 触发条件 | 示例 |
|---------|---------|------|
| 静态阈值 | 指标 > 固定值 | CPU > 80% |
| 动态基线 | 偏离历史均值 > 3σ | 今天 CPU 均值比过去 7 天高 3 倍标准差 |
| 连续突增 | 连续 N 个点上升 | 连续 5 个点 CPU 上升 |
| 突变检测 | 相邻点差值 > 阈值 | CPU 从 30% 突增到 90% |

**异常严重度**
```
Low（低）：     不影响业务，记录即可
Medium（中）：  可能影响性能，创建工单
High（高）：    明显影响业务，立即通知
Critical（紧急）：系统故障，电话+短信
```

**检测频率**
- 指标采集：每 5 秒
- 异常检测：实时（EventEmitter 监听 metric.created）
- 数据保留：最近 7 天原始数据，7 天前聚合为 1 分钟粒度

#### 2.2.3 Agent 自主诊断

**触发机制**
```typescript
@OnEvent('anomaly.detected')
async handleAnomaly(anomaly: Anomaly) {
  // Agent 被异常事件唤醒
}
```

**标准诊断流程（5 步）**
```
Step 1: getServerHealth(serverId)
        ↓ 获取当前实时状态

Step 2: getMetricHistory(serverId, metricType, 30min)
        ↓ 查看历史趋势

Step 3: queryHistoricalAnomalies(serverId, type)
        ↓ 检索相似案例

Step 4: generateDiagnosis(data)
        ↓ 综合分析生成诊断报告

Step 5: 决策
        if severity == 'critical':
            createTicket(priority='high')
        elif severity == 'high':
            sendNotification()
        else:
            logOnly()
```

**诊断输出格式**
```markdown
## 异常诊断报告

**异常信息**
- 服务器：srv-001
- 类型：CPU 突增
- 检测时间：2026-06-18 09:23:15
- 严重度：Medium

**当前状态**
- CPU：96%
- 内存：65%
- 温度：28°C
- 主要进程：java (PID 1234, 80% CPU)

**历史趋势**
过去 30 分钟 CPU 从 35% 逐步上升至 96%，
在 09:20 出现突增（30% → 90% in 2min）

**相似案例**
找到 3 条历史相似异常：
1. 2026-06-15 srv-001 同样 java 进程异常，重启后恢复
2. 2026-06-10 srv-002 类似情况，发现是内存泄漏

**诊断结论**
Java 应用可能存在内存泄漏导致 GC 频繁，建议：
1. 重启 java 进程
2. 检查应用日志
3. 若频繁复发，需排查代码

**处置建议**
创建 Medium 优先级工单，通知运维处理
```

**性能要求**
- P95 诊断延迟：< 30s
- 工具调用成功率：> 95%
- 诊断准确率：> 80%（人工验证）

#### 2.2.4 WebSocket 实时推送

**推送事件**

| 事件名 | 数据结构 | 推送频率 |
|-------|---------|---------|
| `metrics` | `{ serverId, metricType, value, timestamp }[]` | 每 5s |
| `anomaly` | `{ id, serverId, type, severity, detectedAt }` | 实时 |
| `diagnosis` | `{ anomalyId, conclusion, steps, action }` | 实时 |
| `status` | `{ status: 'diagnosing' \| 'resolved' }` | 实时 |

**连接管理**
- JWT 鉴权：连接时验证 token
- 心跳检测：30s 一次 ping/pong
- 断线重连：指数退避（1s → 2s → 4s → 8s）
- 房间隔离：每个用户只收到自己权限内的设备数据

#### 2.2.5 异常列表

**列表字段**
- 异常标识：红/黄/橙色点 + 类型图标
- 服务器名称：可点击跳转详情
- 异常类型：CPU 突增 / 温度异常 / 内存泄漏...
- 检测时间：相对时间（3 分钟前）
- 状态：open / diagnosing / resolved / ignored
- 操作：查看详情 / 手动处理 / 忽略

**状态流转**
```
open → diagnosing → resolved
  ↓
ignored
```

**排序规则**
1. 未解决异常优先
2. 严重度高优先
3. 时间倒序

### 3.2 数据量估算

**单设备指标量**
- 4 类指标 × 每 5s 一次 = 48 条/分钟 = 69,120 条/天
- 10 台设备 = 691,200 条/天
- 保留 7 天 = 约 480 万条

**异常量**
- 假设每台设备每天 5 个异常 = 50 个/天
- 保留 30 天 = 1,500 条

**诊断记录**
- 与异常 1:1，1,500 条

**工具调用记录**
- 每次诊断平均 5 个工具 = 7,500 条

### 3.3 数据保留策略

| 数据类型 | 保留策略 |
|---------|---------|
| 原始指标 | 7 天 |
| 聚合指标（1min） | 30 天 |
| 聚合指标（1h） | 365 天 |
| 异常记录 | 永久 |
| 诊断记录 | 永久 |
| 工具调用记录 | 90 天 |

## 4. 技术方案

### 4.1 技术选型理由

| 技术 | 选型理由 |
|------|---------|
| NestJS | 企业级 Node 框架，模块化架构，适合学习后端思维 |
| Prisma | TS 友好 ORM，迁移管理方便，类型安全 |
| Postgres | 成熟稳定，支持 JSON/时序数据，pgvector 扩展为 P2 做准备 |
| WebSocket | 实时推送必需，socket.io 生态成熟 |
| Vercel AI SDK | TS 原生，API 简洁，Tool Calling 支持好 |
| DeepSeek | 性价比高，中文支持好，Tool Calling 稳定 |

### 4.2 核心技术难点

#### 难点 1：实时数据流性能
**问题**：10 台设备 × 4 指标 × 5s = 8 条/秒，长时间运行前端可能卡顿

**解决方案**：
- WebSocket 数据压缩
- Echarts dataZoom + sampling
- 前端数据只保留最近 5 分钟
- 滑动窗口删除旧数据

#### 难点 2：Agent 诊断延迟
**问题**：多步工具调用可能超过 30s

**解决方案**：
- 并行调用不依赖的工具
- Prompt Cache 减少首字节延迟
- 设置 maxSteps=8 避免无限循环
- 超时自动降级为简单诊断

#### 难点 3：异常检测准确率
**问题**：静态阈值误报率高

**解决方案**：
- 动态基线 + 历史统计
- 连续多点确认
- 用户反馈循环（误报可标记）
